## 1. Problem Statement
The product's promise is "pinged on your phone" before a deadline bites, but
there is no way for a browser to receive a push yet. Without a stored push
subscription per user/device, the planned notification cron has no one to send
to. This story builds the subscription handshake so later stories can actually
deliver alerts.

## 2. Goals
- Let a signed-in user opt in to push and register a `PushSubscription`.
- Persist subscriptions per user (a user may have several devices).
- Send a test push so the user confirms it works.
- Let a user revoke notifications and prune dead subscriptions.

## 3. Non-Goals
- Scheduling or urgency logic (story 2.2).
- Email or SMS channels.
- Notification preferences/quiet hours (later story).

## 4. User Stories
- **As a student**, I want to enable browser notifications so that I get pinged before deadlines.
- **As a student**, I want a test notification so that I know it's working.
- **As a student**, I want to turn notifications off so that I stop receiving them.

## 5. Functional Requirements
- **FR-1.** The frontend MUST register a service worker and request notification permission only on explicit user action.
- **FR-2.** On grant, the system MUST create a `PushSubscription` with the app's VAPID public key and POST it to the backend.
- **FR-3.** The backend MUST store the subscription (endpoint + keys) against the authenticated `user_id`, de-duplicated by endpoint.
- **FR-4.** The system MUST send a test push via the Web Push protocol after subscribing.
- **FR-5.** On a `410 Gone` / `404` from the push service, the system MUST delete the dead subscription.
- **FR-6.** The system MUST let a user unsubscribe, deleting their subscription(s).

## 6. Non-Functional Requirements
- **Performance** — subscribe round-trip p95 < 1.5s; test push dispatched async.
- **Security** — VAPID **private** key only on the server via `VAPID_PRIVATE_KEY` env var; public key shipped to client; per-user authz on all routes.
- **Privacy** — store only the push endpoint and keys; no message history retained beyond send logs.
- **Accessibility** — the enable/disable control is a labelled toggle; status announced via `aria-live`.
- **Reliability** — sends are idempotent per subscription; dead-endpoint pruning keeps the table clean.
- **Observability** — log `push_subscribe`, `push_test_sent`, `push_pruned` with user id and subscription id (never keys).

## 7. Acceptance Criteria
- **AC-1.** *Given* a signed-in user who grants permission, *When* they enable notifications, *Then* a subscription row is stored and a test push arrives.
- **AC-2.** *Given* a stored subscription that returns `410`, *When* a send is attempted, *Then* the row is deleted.
- **AC-3.** *Given* a subscribed user, *When* they disable notifications, *Then* their subscriptions are removed and no further pushes are sent.
- **AC-4.** *Given* an unauthenticated request to a push route, *When* made, *Then* it returns `401`.

- Unique index on `endpoint`; index on `user_id`.
- RLS: users read/write only their own rows.
- Migration: `supabase/migrations/004_push_subscriptions.sql`.

## 9. API Surface
- `POST /api/push/subscribe` — auth required. Body = serialized `PushSubscription`. Upserts by endpoint, sends test push.
- `DELETE /api/push/subscribe` — auth required. Body `{ endpoint }` (or all for user). Deletes.
- `GET /api/push/vapid-public-key` — returns the public key for client subscription.

## 14. Risks & Mitigations
| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| iOS Safari push quirks (PWA install required) | M | M | Detect support; show install hint; document limitation |
| Stale subscriptions accumulate | M | L | Prune on `404`/`410` during send |
| VAPID private key leak | L | H | Server-only env var; never shipped to client; rotate runbook |

## 15. Rollout Plan
- Flag: `feature.web_push` (default **off**).
- Sequence: migration → deploy VAPID + routes → deploy service worker + toggle → dogfood test push → GA.
- Rollback: flag off; existing subscriptions retained but unused.

## 16. Test Plan
- **Unit** — subscription serialization, dead-endpoint pruning logic.
- **Integration** — subscribe stores row + sends test (mocked push service); `410` prunes; authz `401`.
- **E2E (Playwright)** — enable → test arrives (mocked SW); disable → row removed.
- **Security** — private VAPID key never in client bundle or responses.
- **Manual** — real device subscribe + receive across Chrome/Firefox; note iOS behavior.
